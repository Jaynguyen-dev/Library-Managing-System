## 🎯 Project Overview

**LibraryLMS** is a monolithic web application built for the **SE104 (Introduction to Software Engineering)** course at the **University of Information Technology (UIT)**. It digitizes the day-to-day operations of a university or community library, including account management, book catalogue control, borrow/return transactions, automated fine handling, wallet top-ups, reservations, and real-time dashboard analytics.

### Target Users & Roles
*   **Librarian (Staff/Admin):** Has access to the main **Staff Dashboard**. Responsible for book catalogue CRUD operations, member management, issuing borrows, processing returns, managing fines, and reviewing crawl logs.
*   **Reader (Student/Patron):** Authenticated as `user`. Has access to the **Reader Dashboard**. Can search the catalogue, check borrow history, view in-app notifications, maintain a digital wallet, top up funds, pay overdue fines, and reserve books.

## 🏗️ System Architecture

The application implements a classic **three-tier monolithic architecture**, combining frontend representation, server application logic, and data persistence layers in a tightly integrated system.

```mermaid
graph TD
    subgraph Client [React SPA - Port 5173]
        UI[Spotify-Inspired Dark UI]
        State[Context API & Axios Interceptors]
    end

    subgraph Server [Express.js API Server - Port 3001]
        R[API Routers]
        M[Auth & Role Guard Middlewares]
        C[Controllers]
        S[Business Services]
        CR[Crawl Pipeline]
    end

    subgraph Database [SQL Database]
        DB[(Microsoft SQL Server)]
    end

    subgraph External APIs [Book Metadata Sources]
        OL[Open Library REST API]
        GB[Google Books API]
        VB[Vinabook Web Scraper]
    end

    UI <-->|HTTP / JWT JSON| R
    R --> M --> C --> S
    S <-->|Prisma ORM| DB
    CR <-->|Fetch / Scrape| External APIs
    CR -->|Enrich| DB
    C --> CR
```

### Request-Response Lifecycle
1.  **Client Request:** The React client fires an API request via `axios`. A request interceptor automatically embeds the JWT in the `Authorization: Bearer <token>` header.
2.  **Routing & Security:** Express maps the route. The `authMiddleware` validates the token, and the `roleGuard` verifies permissions (e.g., rejecting students trying to access `/api/billing`).
3.  **Validation:** `express-validator` checks query params and request bodies, ensuring correct formats (e.g. valid ISBN-13 format).
4.  **Service Delegation:** Controllers remain thin, delegating core business operations (like calculating overdue charges or evaluating queue priorities) to separate services.
5.  **ORM / Database:** Prisma executes queries against the **Microsoft SQL Server** backend.
6.  **Structured Response:** The controller returns a standardized JSON object:
    ```json
    { "success": true, "data": { ... }, "message": "Operation successful" }
    ```

## 🎨 Aesthetics & Design System

The application layout is built on a custom **Spotify-inspired dark theme** design system, defined via CSS variables in [index.css](file:///client/src/index.css):

*   **Dark Immersive Canvas:** Main backgrounds use deep charcoal and blacks (`#121212`, `#181818`, `#000000`). This ensures page layout elements disappear in the background, allowing book cover artwork to supply dynamic color.
*   **Spotify Green Accent:** Accent elements, active navigation items, badges, and primary buttons use **Spotify Green (`#1ed760`)** for a crisp, tactile call to action.
*   **Pill & Circle Geometry:** Controls use extreme rounded radii (pills of `500px` to `9999px` for search bars and buttons; `50%` circles for user avatars, play/pause controls, and action buttons).
*   **Dense Information Layout:** Dense content layouts over excessive margins, mirroring modern dashboard media controllers.
*   **Elevation & Depth:** Uses high-opacity shadows (`rgba(0,0,0,0.5) 0px 8px 24px`) to create clear layered depth on dialog blocks, slide-out components, and hover card actions.

## 🗄️ Database Schema

The database uses third-normal form (3NF) relational models mapping entities in [schema.prisma](file:///server/prisma/schema.prisma):

```
+--------------+        +-----------------+        +---------------+
|     User     |------< |  BorrowRecord   |------< |  BorrowItem   |
| (Librarian/  |        | (header status) |        | (book quantity|
|  Reader)     |        +--------+--------+        +-------+-------+
+------+-------+                 |                         |
       |                         |                         |
       |                         |                         v
       |                         |                 +---------------+
       |                         |                 |     Book      |
       |                         |                 |  (catalogue)  |
       |                         |                 +-------+-------+
       |                         |                         |
       v                         v                         v 1:1
+------+-------+        +--------+--------+        +-------+-------+
|    Wallet    |------< |      Fine       |        |  BookMetadata |
| (balance and |        | (overdue penalty|        | (cover image, |
| transactions)|        |  2,000 VND/day) |        | description)  |
+--------------+        +-----------------+        +---------------+
```

### Models Summary:
1.  **User:** Stores credential details, role string (`librarian`, `user`), and activation state.
2.  **Book:** Tracks catalog details, ISBN-13 index, and quantities (`total_quantity` and `available_quantity`).
3.  **BookMetadata:** Holds crawled enrichment data (cover image URLs, publisher metadata, descriptions, page count, ratings, and crawler sources).
4.  **BorrowRecord & BorrowItem:** Employs header-detail splitting to document library borrows (allowing multi-book transactions, tracking due date, return date, and return request triggers).
5.  **Fine & FinePayment:** Handles dynamic calculation of fines (overdue days × 2,000 VND).
6.  **Wallet & WalletTransaction:** Digital wallet accounts for students to top up, request refunds, or clear unpaid fines.
7.  **Reservation:** Book queues managing waiting lines for popular, checked-out volumes.
8.  **Notification:** Dispatches transactional info (overdue notices, checkout reminders) directly to readers.
9.  **CrawlLog:** Stores diagnostic run histories for API crawler jobs.

## ⚡ Core Features

### 1. Advanced Real-Time Dashboard
*   **Librarian Dashboard:** Summarizes total catalog volumes, active system users, active checkouts, overdue borrows, and total outstanding fine metrics. Includes detailed listings of overdue accounts.
*   **Reader Dashboard:** Custom layout showing borrowed volumes, upcoming due-dates, wallet status, active reservation list, and system alerts.

### 2. Multi-Item Borrow & Return Transactions
*   Allows librarians to check out up to 3 books in a single transaction.
*   Enforces inventory control—available inventory balances auto-decrement on checkout and restore on return.
*   Includes "Return Request" hooks where readers can request returns, and staff confirm the completion.

### 3. Integrated Wallet & Fine Payment
*   Overdue book fines calculate automatically upon check-in at a rate of **2,000 VND per day**.
*   Readers can top up their digital wallet balance and pay fines in a single click, recording payments in a transparent audit ledger.

### 4. Metadata Crawl Pipeline
*   **Auto-crawl on Create:** Saving a new book with its ISBN-13 triggers the background crawler.
*   **Multi-Source Merging:** Merges data from Open Library REST API, Google Books API, and Vinabook web scraping (using Cheerio).
*   **Batch Ingestion:** Staff can queue batch-enrichment runs for all catalog entries lacking metadata, using concurrency-limited polite request gaps to prevent rate limiting.

### 5. Reservation Queues
*   Allows users to reserve books when `available_quantity == 0`.
*   Maintains sequential queue order. When a copy returns, the front-of-queue user is notified and holds a priority reserve period.

## ⚙️ Installation & Setup

### Prerequisites
*   [Node.js](https://nodejs.org/) (v18 or higher)
*   [Microsoft SQL Server](https://www.microsoft.com/en-us/sql-server/)

### 1. Database Setup
Ensure MS SQL Server is running, then configure a database (e.g. `library_db`).

### 2. Backend Installation
1.  Navigate to the server directory:
    ```bash
    cd server
    ```
2.  Install dependencies:
    ```bash
    npm install
    ```
3.  Configure environment variables. Copy `.env.example` to `.env` and fill in your connection details:
    ```env
    PORT=3001
    DATABASE_URL="sqlserver://localhost:1433;database=library_db;user=SA;password=YourPassword;encrypt=true;trustServerCertificate=true;"
    JWT_SECRET="generate_a_long_cryptographic_secret_key"
    CLIENT_URL="http://localhost:5173"
    ```
4.  Generate Prisma client and apply migrations:
    ```bash
    npx prisma migrate dev --name init
    ```
5.  Seed the database with sample data:
    ```bash
    npx prisma db seed
    ```
6.  Start the Express server:
    ```bash
    npm run dev
    ```

### 3. Frontend Installation
1.  Navigate to the client directory:
    ```bash
    cd ../client
    ```
2.  Install dependencies:
    ```bash
    npm install
    ```
3.  Start the local dev server:
    ```bash
    npm run dev
    ```
4.  Open [http://localhost:5173](http://localhost:5173) in your browser.

## 🔌 API Endpoint Reference

### 🔐 Authentication (`/api/auth`)
*   `POST /api/auth/register` - Registers a reader account.
*   `POST /api/auth/login` - Authenticates user; returns JWT and user profile.
*   `GET /api/auth/me` - Resolves details of the currently signed-in user.

### 📚 Catalog Management (`/api/books`)
*   `GET /api/books` - Lists catalog items (supports search queries).
*   `GET /api/books/:id` - Detailed profile of a specific book.
*   `POST /api/books` - Adds a new book (triggers crawler background task).
*   `PUT /api/books/:id` - Updates text details of an catalog entry.
*   `DELETE /api/books/:id` - Soft deletes a book (setting `is_deleted = true`).

### 👥 Member Management (`/api/users`)
*   `GET /api/users` - Fetches members (optional role filters).
*   `POST /api/users` - Creates accounts (Librarian only).
*   `PUT /api/users/:id` - Modifies basic member parameters.
*   `PATCH /api/users/:id/toggle-active` - Locks or unlocks a user's account.

### 🔄 Borrows & Returns (`/api/borrows`)
*   `POST /api/borrows` - Creates a checkout record (max 3 books; decrements inventory).
*   `GET /api/borrows` - Lists borrows (Librarian only; supports status filters).
*   `GET /api/borrows/my` - Returns logged-in reader's borrowing history.
*   `PATCH /api/borrows/:id/return` - Confirms return, increments inventory, and calculates overdue fines.

### 💳 Digital Wallets (`/api/wallet`)
*   `GET /api/wallet` - Fetches reader's wallet balance and transactions list.
*   `POST /api/wallet/topup` - Adds money to the user's wallet.
*   `POST /api/wallet/pay-fine` - Pays a fine using wallet funds.

### ⏳ Book Reservations (`/api/reservations`)
*   `POST /api/reservations` - Joins a book's reservation queue.
*   `GET /api/reservations/my` - Shows current user's reservations.
*   `DELETE /api/reservations/:id` - Cancels an active reservation.

## 🛠️ Architecture Decision Records & Implementation Rules

All contributors must adhere to the rules outlined in [IMPLEMENTATION_RULES.md](file:///IMPLEMENTATION_RULES.md):

1.  **JavaScript Only:** Never introduce TypeScript into this project.
2.  **Strict Database Enforcement:** Use Microsoft SQL Server (MSSQL) as the exclusive database engine. Prisma's database provider must remain `sqlserver` (PostgreSQL and MySQL are strictly prohibited).
3.  **Encapsulated Logic:** Put business rules (e.g. fine rate calculations or crawler data mapping) in **Services**. Controllers must remain thin, handling only input validation and JSON response mapping.
4.  **Preserved API Interfaces:** Never alter seeded API response shapes or JSON structures, ensuring frontend-backend compatibility.
5.  **Achromatic UI Hierarchy:** Maintain the Spotify-inspired theme rules outlined in [DESIGN.md](file:///DESIGN.md). Ensure gray borders are replaced by shadow elevation, and keep typography dense.
